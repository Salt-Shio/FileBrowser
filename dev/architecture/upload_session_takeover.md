# 架構決策紀錄 (ADR): 分塊上傳死鎖與並發防禦機制

**日期**: 2026-06-24
**主題**: 從 Redis Lock 到 Fencing Token (Session Takeover) 的架構演進

## 1. 問題的發現 (The Deadlock Discovery)

在大型分塊上傳系統中，我們遭遇到了一個嚴重的邊界案例 (Edge Case)：
* **情境**：使用者在上傳到一半時，不小心清除了瀏覽器的 LocalStorage（或強制關閉了不具備快取持久化的無痕視窗），導致前端失去了原本的 `upload_id`。
* **行為**：當使用者再次將同一份檔案拖入視窗時，前端會發起一次全新的 `init_upload` 請求。
* **死鎖 (Deadlock) 發生**：後端為了防禦多裝置同時上傳造成的檔案毀損，會在資料庫尋找同名、同大小的 `conflict_session`。一旦找到，就立刻拋出 `HTTP 400 Conflict: 同名檔案已在上傳中`。
* **最終結果**：舊的視窗已經死亡，新的視窗被後端拒絕，這份檔案陷入了永無止境的「死鎖」，使用者永遠無法繼續上傳這份檔案。

## 2. 方案推演與痛點分析 (Analysis of Failed Approaches)

為了解決這個死鎖，我們曾推演了以下幾種防禦架構，但最終都被否決：

### 方案 A：強制覆蓋 (Force Overwrite)
當遇到衝突時，直接刪除舊的 Session，創建新的 Session 讓其從 0% 重新開始。
* **痛點**：雖然解決了死鎖，但如果使用者是真的「雙開瀏覽器 (同時傳同一個檔案)」，這會導致兩邊的分頁瘋狂互相刪除對方的 Session，引發更嚴重的覆寫崩潰。

### 方案 B：Redis 分散式鎖 (Collaborative Parallel Upload)
允許多個分頁/裝置「共存」並「協作」上傳同一份檔案。在 `upload_chunk` 時使用 `vfs:lock:chunk:{id}` 來避免對同一個分塊的並發寫入，並設定 60 秒 TTL。
* **痛點**：這是一個極度的**過度工程 (Over-engineering)**。
  * 如果客戶端當機，留下的「幽靈鎖 (Ghost Lock)」會導致其他客戶端卡死 60 秒無法重試。
  * 引入了龐大的分散式狀態追蹤，後續如果要實作「限制單一使用者最大並發上傳數」會變得異常困難。

## 3. 靈感與破局思路 (The Breakthrough: Session Takeover)

與其在「多端同時寫入」時去建立龐大的防護網，不如從根本上**「消滅多端並發」**。
架構設計的核心思想轉變為：**「一份上傳進度，同一時間只能有一把合法的鑰匙 (Fencing Token)。」**

這在分散式系統中被稱為 **Session Takeover (會話劫持/接管)** 或 **Fencing Token** 機制。

### 實作機制：
1. **發放 Token**：在 `UploadSession` 資料表中新增一個 `upload_token` 欄位。前端 `init_upload` 會取得 `upload_id` 與這把鑰匙。
2. **驗證 Token**：所有的 `upload_chunk` 與 `finalize_upload` 請求，都必須攜帶這把 `upload_token`。
3. **合法劫持 (Hijack)**：當 `init_upload` 再次遇到 `conflict_session` 時，**後端不拋錯阻擋，而是直接更新該會話的 `upload_token` (換鎖)**，並將目前的進度回傳給新客戶端。

### 如何完美解決所有問題？
* **解決死鎖**：清除 LocalStorage 後重新拖入檔案，後端會幫你「換新鎖」並歸還舊進度，無縫續傳！
* **防禦多開併發**：如果使用者惡意雙開，當第二個瀏覽器呼叫 `init_upload` 的瞬間，第一台瀏覽器手上的 Token 立刻淪為廢紙。下一秒它發送分塊時就會吃下 `401 Unauthorized` 錯誤並被系統乾淨俐落地踢下線。

## 4. 極端風險的評估與取捨 (Trade-offs & Edge Cases)

這個極度輕量的架構，在極限的併發測試下，仍有兩個微小的 Race Condition 風險：
1. **飛行中的分塊衝突 (In-Flight Race)**：在換 Token 的那幾十秒 HTTP 傳輸空窗期內，舊瀏覽器與新瀏覽器可能會寫入同一個分塊。
2. **結算劫持 (Finalize Race)**：舊瀏覽器正在執行大檔案合併時被新瀏覽器劫持。

**架構決策**：
經過詳細評估，我們決定**大膽放生**這兩個極端風險，不在應用層加入複雜的 Double Check 迴圈或狀態機鎖。
* **理由 1**：在「單一瀏覽器」的正常生命週期中，前端由迴圈與 `Promise.all` 嚴格控制順序，這兩個問題的發生率是 **0%**。
* **理由 2**：即使在惡意雙開的極端測試下觸發，最壞的情況僅是引發 `PermissionError` (OS 檔案鎖防禦) 或覆寫相同的位元組，導致該次 HTTP 請求噴出 `500 Error`。前端只要透過重試機制即可恢復，**絕對不會造成伺服器崩潰或檔案永久性毀損**。

**結論**：用最少、最乾淨的程式碼，解決 99% 關鍵痛點 (死鎖與雙開防護)，把複雜的極端防禦留在可控的 Exception 與 Retry 機制中，是目前最符合系統穩定性與維護性的優雅決策。
