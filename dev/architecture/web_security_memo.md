# 網路安全與認證機制備忘錄 (Web Security Memo)

這份文件記錄了本專案為何選擇 **OAuth2 Header (Bearer Token)** 而非 Cookie 認證的深層原因，以及相關安全機制的運作原理。

## 1. 核心名詞定義

| 名詞 | 全稱 | 核心邏輯 |
| :--- | :--- | :--- |
| **SOP** | Same-Origin Policy (同源政策) | **瀏覽器的預設鎖**。禁止 A 網站「讀取」B 網站的資料，但**並不完全阻擋「寫入」或「發送請求」**。這就是 CSRF 攻擊的切入點。 |
| **CSRF** | Cross-Site Request Forgery (跨站請求偽造) | **利用 Cookie 的自動化特性**。既然 SOP 不擋「寫入」，黑客只要誘導瀏覽器「發送」請求即可完成攻擊，不需讀取回傳結果。 |
| **CORS** | Cross-Origin Resource Sharing (跨來源資源共享) | **SOP 的白名單**。由伺服器定義哪些網域可以跨站存取資源。 |
| **JWT** | JSON Web Token | **帶有簽名的數據包**。確保 Token 內容在傳輸過程中不會被竄改。 |

---

## 2. 為什麼 Cookie 認證不安全？ (CSRF 漏洞)

*   **自動化特性**：瀏覽器被設計為對同網域請求「自動帶入 Cookie」。
*   **SOP 的盲點**：SOP 雖然禁止「讀取」跨站回應，但 **並不禁止「發送」** 跨站請求。
*   **攻擊場景**：使用者在登入 A 站的狀態下點進惡意的 B 站，B 站可以發送一個指向 A 站的轉帳請求，瀏覽器會自動帶上 A 站的 Cookie，導致攻擊成功。

---

## 3. 本專案的防禦策略：OAuth2 Header (Bearer Token)

我們透過將 Token 放在自定義 Header (`Authorization: Bearer ...`) 來徹底封鎖 CSRF。

### 防禦層級 A：打破自動化
*   瀏覽器 **絕對不會自動** 在請求中加入自定義 Header。
*   這個 Header 必須由前端程式碼（如 Vue/Axios）「主動」加入。
*   黑客無法透過 `evil.com` 讀取您的 Token（受 SOP 保護），因此他無法偽造正確的 Header。

### 防禦層級 B：強制觸發「預檢請求 (Preflight)」
*   根據瀏覽器規範，帶有「自定義 Header」的請求屬於 **非簡單請求**。
*   瀏覽器會強制先發送一個 `OPTIONS` 請求（Preflight）詢問伺服器。
*   由於黑客的網域不在我們後端的 **CORS 白名單** 內，`OPTIONS` 請求會被拒絕，攻擊請求根本不會發出。

---

### 總結架構決策

1.  **放棄 Cookie**：避開 CSRF 的先天缺陷。
2.  **採用 Header**：利用「主動注入」與「Preflight 機制」提升安全性。
3.  **選用 JWT**：利用「數位簽章」防止資料被竄改。

---

## 4. 架構反思：工具的「命名」與「邏輯」

在實作過程中，我們對 FastAPI 提供的認證工具進行了深層辨析。這涉及到「高度封裝」與「職責分離」的權衡。

### 角色擬人化對比

*   **OAuth2PasswordBearer** (目前的選擇)：
    > 「我是 OAuth2 標準下的 **『密碼模式』**。我預設這個 Token 是使用者拿帳號密碼換來的，所以我 **『必須』** 知道去哪裡換 (tokenUrl)。」
*   **HTTPBearer** (邏輯純粹的替代方案)：
    > 「我不管您的 Token 是怎麼來的，我只管它在 HTTP Header 裡是不是長成 **『Bearer xxx』**。我的工作就是把 xxx 拿出來。」

### 為什麼會覺得「怪」？ (開發者直覺)

我們發現 `OAuth2PasswordBearer` 存在一種 **強烈的耦合 (Tight Coupling)**：
1.  **功能混雜**：它強制把「提取字串的邏輯」與「與邏輯無關的 UI/文件配置 (tokenUrl)」綁在一起。
2.  **強制性**：身為一個安全性工具，它卻不允許開發者「自選」要不要導航路徑，這在邏輯上是不夠純粹的。

### 結論
雖然 `OAuth2PasswordBearer` 的設計讓「程式碼即文件 (Code is Documentation)」的目標得以達成（換來了自動化的 Swagger 登入視窗），但它也犧牲了組件的「中立性」。

在未來的架構演進中，如果我們追求更極致的職責分離（例如 Token 來源多元化時），轉向 **`HTTPBearer`** 將會是更符合邏輯直覺的選擇。
